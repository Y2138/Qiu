# Qiu 极简 Agent 助手系统迭代规划_0310

## Summary
Qiu 需要从 `QiuChat` 重新定义为“极简 Agent 助手”，而不是“带一点 Agent 能力的 Chat 工具”。

本轮规划的核心不是继续扩运行时复杂度，而是完成一次产品重心切换：
- 用户看到的是一个极简、自然、低学习成本的对话助手
- 系统底层默认开启 Agent 基础能力，但尽量不把 `memory / skills / tools / mcp` 这些概念暴露给普通用户
- Agent 能做最基础的规划、循环执行、工具调用、文件读取、偏好记忆
- 高级用户可以通过 `skill` 和未来的扩展机制增强能力，但不让扩展能力污染主界面

目标产品形态不是“全能平台”，而是“默认可用、默认克制、默认可理解”的个人 Agent 助手。

## 背景判断
结合当前已有规划和实现，项目已经具备以下基础：
- 已有独立 `AgentRun / AgentCheckpoint / AgentMemoryEntry` 数据模型
- 已有 `AgentRuntime`、事件流、checkpoint、memoryMode、skills、tool registry 等运行时骨架
- 当前前端已经开始暴露 `AgentRunPanel`、输入区 Agent 配置、memory 管理等概念

但从产品角度看，当前仍存在明显错位：
- 名称上仍偏 `Chat` 产品，不符合“Agent 助手”定位
- 页面结构仍在强调“Agent 功能模块”，而不是“用户任务完成”
- 用户需要理解过多系统概念，认知负担偏高
- Agent 的人格、长期偏好、文件理解等“助手感”能力不够完整
- 执行状态与 chat 流结合还不够自然

因此，本轮应把重点从“更多能力”切到“更正确的产品表达”。

## 产品定位
### 新定位
`Qiu = 你的 Agent 助手`

它的职责不是做一个复杂工作台，而是：
- 接收用户目标
- 在对话中拆解任务并逐步执行
- 必要时调用工具、读取文件、保存偏好
- 在用户几乎无感的情况下完成一次最小闭环

### 目标用户
- 希望用自然语言完成日常任务的普通用户
- 希望有“一个有风格、记得我、能帮我做事”的轻量 Agent 助手的用户
- 少量希望自定义 skill/tool 的进阶用户

### 核心价值
- 极简：界面和概念都要少
- 可理解：用户始终知道它在做什么
- 可延展：先支持最基本的扩展能力，不做平台化过度设计

## 产品原则
### 1. Agent 默认存在，但不默认被看见
- `session memory`、`skills`、基础 `tool call` 默认开启
- 普通用户不需要理解这些名词，也不需要在主界面手动配置
- 页面默认只保留输入框、消息流、必要的执行状态反馈

### 2. 把“配置”从聊天主流程剥离到设置页
- Agent 角色偏好、语气、记忆策略、扩展能力管理统一收敛到设置页
- 聊天页不再承担“Agent 开关面板”的职责
- 用户只有在“我要调教助手”时才进入设置

### 3. Agent 状态应嵌入对话，而不是旁路展示
- `Agent Run Panel` 不作为主展示结构存在
- Agent 当前执行的步骤、工具调用、暂停/继续、文件读取，应直接体现在消息流中
- 用户理解的是“助手正在做什么”，不是“某个运行面板里有什么事件”

### 4. 记忆要有感知结果，无需暴露底层机制
- 用户只需要感知“它记住了我的偏好”
- 用户不需要在聊天页理解 `session/user memory` 区别
- 只有在设置页中，才以较低频的方式提供查看、删除、关闭等控制

### 5. 先做可信的单 Agent，再谈多 Agent 和 MCP
- 本期只做单 Agent 主线
- 先保证基本规划、loop、tool call、文件读取、记忆和偏好稳定可用
- 用户自定义扩展优先考虑 `skill`；`MCP` 若实现成本或稳定性不足，可以暂缓

## 目标体验
用户打开 Qiu 后，应当感受到：
- 这是一个会主动思考和执行的助手，而不是普通聊天框
- 它有稳定人格和风格，回复方式可被调教
- 它会在对话中自然展示“正在分析 / 正在读取文件 / 正在执行第 2 步”
- 它默认能记住一些长期偏好，但不会过度打扰
- 它看起来很简单，但关键时刻能完成最基本的 Agent 行为

## 用户故事
### A. 首次使用
- 作为一个第一次进入产品的用户，我希望看到的是一个非常简单的对话页，这样我不用先学习一堆 Agent 概念。
- 作为一个第一次使用的用户，我希望我直接输入需求后，系统就能自动进入 Agent 模式，而不是先让我决定是否开启 Agent。
- 作为一个新用户，我希望当 Agent 正在做事时，它能在对话里提示当前进度，这样我不会误以为系统卡住了。

### B. 日常对话与任务执行
- 作为一个用户，我希望输入“帮我整理一下这份需求并给出下一步计划”后，Qiu 能自动进行简单规划，而不是只生成一段泛泛建议。
- 作为一个用户，我希望当 Agent 调用了工具、读取了文件或进入下一步时，系统在聊天流中告诉我它做了什么。
- 作为一个用户，我希望任务暂停或失败时，能在当前对话中直接继续，而不需要理解 checkpoint、run panel 这些术语。

### C. 个性化助手
- 作为一个用户，我希望可以在设置里定义 Agent 的语气和风格，例如“温柔、坚定、凌厉、简洁”，这样这个助手更像“我的助手”。
- 作为一个用户，我希望 Qiu 能逐步记住我的常用偏好，例如我喜欢的表达方式、工作方式和常见上下文，而不是每次都重复说明。
- 作为一个用户，我希望记忆是可控的，当我不想让系统保留某些内容时，可以去设置里删除。

### D. 文件工作流
- 作为一个用户，我希望可以上传文档、截图或文本文件，并让 Agent 读取内容后继续工作。
- 作为一个用户，我希望文件上传后的行为是自然的，例如“已读取文件，正在提取重点”，而不是把文件功能做成独立复杂模块。

### E. 进阶扩展
- 作为一个进阶用户，我希望未来可以扩展 Qiu 的工具能力，例如增加自定义 skill。
- 作为一个进阶用户，我希望这些扩展能力不会干扰普通使用，但需要时可以配置。

## Agent 故事
### A. Agent 角色与系统提示词
Agent 不应只是一个固定“助手 prompt”，而应由稳定系统人格和可配置偏好共同决定。

Agent 系统提示应至少由以下部分组成：
- `Base Identity`
  - 你是 Qiu，用户的个人 Agent 助手
  - 目标是理解任务、必要时规划、逐步执行，并在过程里保持状态透明
- `Behavior Policy`
  - 默认优先帮助用户推进任务，而不是长篇解释
  - 遇到可执行任务时，应先给出最小计划，再执行
  - 遇到不确定或风险高的操作时，应先确认
- `Communication Style`
  - 基于用户偏好注入语气，如温柔、专业、凌厉、简洁
  - 风格只影响表达，不影响事实性和安全边界
- `Memory Policy`
  - 自动使用会话记忆维持上下文
  - 仅把明确、稳定、低风险的长期偏好写入自动化记忆
- `Tool Use Policy`
  - 当文件、检索、结构化处理等工具能更好完成任务时，应主动调用
  - 调用前后要在对话流中产出可理解状态
- `Loop Policy`
  - 支持基础的 `plan -> act -> observe -> continue / stop`
  - 控制步数，避免无意义循环

### B. Agent 必备系统能力
- 能维护当前目标
- 能做最小规划
- 能根据观察结果调整下一步
- 能调用基础工具
- 能读取用户上传文件
- 能在长对话中保留关键上下文
- 能记住用户长期偏好
- 能把执行状态翻译成用户可理解的话

### C. Agent 不必在本期拥有的能力
- 复杂多 Agent 协作
- 开放式 MCP 市场
- 高自治、长时间后台运行
- 复杂 workflow builder
- 用户手工编排精细 prompt / tool graph

## 核心产品改造项
### 1. 品牌与信息架构重构
- 产品名称从 `QiuChat` 调整为 `Qiu`
- 页面文案、空态、欢迎语、导航命名全部从“聊天工具”转向“Agent 助手”
- 核心信息架构收敛为：
  - 聊天页
  - 历史会话
  - 设置页
- 不再把 `Agent` 作为一个显式一级工作区概念

### 2. 聊天页极简化
- 移除默认展示的 Agent 面板、Agent 开关、运行配置区
- 输入区仅保留必要操作：输入、发送、上传文件
- 默认能力直接生效：
  - session memory
  - 基础 skill policy
  - 基础 tool call
- 如确需展示状态，只以内嵌消息块呈现：
  - 正在理解任务
  - 正在执行第 1 步
  - 正在读取文件
  - 已完成，正在总结

### 3. 对话流中的 Agent 状态表达
- 将当前执行状态直接嵌入消息流，不再依赖单独 `Agent Run Panel`
- 事件对用户的表达应翻译为自然语言状态：
  - `run_started` -> “开始处理你的请求”
  - `plan_snapshot` -> “已拆分为 3 个步骤”
  - `tool_call` -> “正在读取文件/调用工具”
  - `memory_compacted` -> 不主动暴露底层术语，可弱化为“已整理上下文”
  - `run_paused` -> “执行中断，可继续”
  - `final` -> “已完成”
- 用户能在对话中看到“当前步骤”，但不应被大量技术事件淹没

### 4. Agent 偏好设置页
- 设置页新增 `Agent 偏好` 模块
- 本期建议支持的可配置项：
  - 回复语气：温柔 / 专业 / 凌厉 / 简洁
  - 回复密度：简短 / 平衡 / 详细
  - 工作方式：默认先规划 / 默认直接执行
  - 自动记忆：开启 / 关闭
  - 扩展能力：skills 管理入口
- 设置页应同时承担：
  - 查看长期记忆条目
  - 删除不需要的自动记忆
  - 管理用户安装或启用的 skill

### 5. 自动化记忆
- 自动化记忆本期只做“偏好型长期记忆”
- 适合自动记住的内容：
  - 语气偏好
  - 常用输出风格
  - 持续稳定的项目背景
  - 常见工作习惯
- 不应自动写入的内容：
  - 一次性任务细节
  - 敏感信息
  - 模型推断但未经用户确认的结论
- 对用户的呈现原则：
  - 聊天中默认不打扰
  - 设置页中可查看、可删除、可关闭

### 6. 文件上传与文件读取
- 聊天输入区支持上传文件
- 上传后消息流应出现明确状态：
  - 文件上传中
  - 文件已读取
  - 正在提取关键信息
- Agent 读取文件应成为标准工具能力，不作为特殊模式处理
- 本期优先支持：
  - 文本文件
  - markdown
  - pdf
  - 常见图片 OCR 或图片理解可作为后续能力

### 7. Markdown 表格展示修复
- 表格渲染属于高频基础体验问题，应作为本期明确修复项
- 目标不是只“能显示”，而是保证：
  - 小屏可横向滚动
  - 表头、边框、单元格间距清晰
  - 中英文混排时不挤压变形
  - 代码块、列表、表格在同一消息内不互相破坏布局

### 8. 扩展能力策略
- 本期支持“用户可扩展 skill”的产品预留
- 扩展原则：
  - 普通用户默认无感
  - 进阶用户在设置中管理
  - 扩展后仍遵守统一 Agent 行为边界
- `MCP` 暂时不作为主规划要求，原因：
  - 实现复杂度和连接稳定性风险偏高
  - 当前产品阶段更需要先把基础助手体验打稳
- 若后续纳入，建议仅作为进阶实验能力入口

## 推荐的系统能力分层
### L1. 默认基础层
- 单 Agent
- 基础 system prompt
- session memory
- 基础 tool call
- 基础 loop
- 文件读取

### L2. 个性化层
- 用户语气偏好
- 回复风格偏好
- 自动化长期记忆
- 记忆管理

### L3. 扩展层
- skill 开关与管理
- 用户安装自定义 skill
- MCP 预留接口（可暂不开放）

## 分阶段迭代建议
### Phase 1. 产品重定位与界面收敛
- 品牌从 `QiuChat` 调整为 `Qiu`
- 移除主界面中的显式 Agent 面板和复杂开关
- 将 Agent 状态嵌入消息流
- 修复 markdown 表格展示

验收标准：
- 新用户进入首页后，无需学习 Agent 配置即可开始使用
- 用户能从聊天流中理解当前执行状态
- 主界面不再出现冗余 Agent 配置区和 Run Panel

### Phase 2. 个性化与自动记忆
- 上线 Agent 偏好设置页
- 支持语气、回复密度、工作方式设置
- 支持自动化长期记忆的开启、查看、删除

验收标准：
- 用户能显式感知“这个助手有风格，而且会记住我”
- 用户可以管理长期记忆，且操作成本低

### Phase 3. 文件工作流与基础扩展
- 完善文件上传和 Agent 读取链路
- 在设置页开放 skill 管理入口
- 形成“默认简单，进阶可扩展”的双层产品结构

验收标准：
- 用户上传文件后，Agent 能自然接续处理
- skill 扩展不影响普通用户主流程

## 关键验收问题
文档落地后，应持续用以下问题检验方案是否正确：
- 普通用户第一次使用时，是否会把 Qiu 当成“复杂工具”而不是“可直接使用的助手”？
- 用户是否能在不理解 Agent 专业名词的情况下知道系统在做什么？
- 偏好设置是否足够少，但又足够形成“我的助手”感？
- 自动记忆是否真正提升体验，而不是制造隐私和误记忆风险？
- 文件读取是否是自然工作流，而不是附加功能入口？
- skill 扩展是否只服务进阶用户，而没有污染默认界面？

## 本期明确不做
- 多 Agent 编排产品化
- 复杂 Agent 工作台
- MCP 大规模接入
- 面向普通用户暴露大量 runtime 概念
- 复杂自动化编排

## 结论
Qiu 下一阶段最重要的不是“把 Agent 做得更重”，而是“把 Agent 做得更自然”。

正确方向应是：
- 底层保留 Agent runtime、memory、skills、tool call 能力
- 上层把这些能力折叠进一个极简、低打扰、可理解的助手体验
- 让用户感受到的是“Qiu 在帮我做事”，而不是“我在操作一个 Agent 系统”

这会比继续堆更多面板、更多概念、更多可见配置，更接近一个真正可用的个人 Agent 助手。
